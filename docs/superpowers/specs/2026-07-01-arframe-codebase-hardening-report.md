# Codebase Hardening Report — explorer/explorerengine → arframe/arpillar

Oracle for the arframe UI-redesign implementation plan. All facts below were re-verified against source (not taken on the audit's word); citations are `file:line`.

## 1. Verdict

The current codebase is **substantially ready to become arpillar+arframe at the engine layer and structurally hostile at the app layer**. The engine spine — the S7 document tree (`aaa_class.R:95-255`), the total `render_plan()` seam (`render_plan.R:65`), the versioned-JSON serializer, the DuckDB pushdown catalog, and the cards/tabular render tail — is clean, tested, Shiny-free, and carries to arpillar near-verbatim. The three biggest risks are all real and confirmed in source: (1) **the model carries no dataset binding anywhere** (`render_plan(catalog, id, object, filters)` threads the dataset id as a *call argument* at `render_plan.R:65`), so a serialized report is not self-contained and cannot reopen-and-re-render — this is a schema change that forces a real migrator; (2) **the spec's `report→page→output` topology does not exist** (the model is `report→task→object`, no page level — `aaa_class.R:243-255, 217, 193`), a non-additive schema change the current no-op migrator (`serialize.R:126-129`) cannot express; (3) **`setup_pane.R` is a 7085-line module whose 942-line `do_commit()` (`setup_pane.R:3908-4849`) is the entire options-shaping contract**, and it must be split four ways across the dual rail with no declarative schema to split against. The redesign lives or dies on landing a **two-stage cached render seam** and a **typed options schema** before touching the UI frame — without them, direct-manipulation "screen==paper" will be visibly laggy and the four-rail commit split will be unbuildable.

## 3. Source-grounded constraints the plan MUST honor

These are verified facts. The redesign cannot contradict them.

1. **`render_plan()` seam shape is `render_plan(catalog, id, object, filters = list())`** (`render_plan.R:65`). The dataset id is a **call argument, not a model property** — confirmed: `id` is threaded into `.require_entry(catalog, id)` at render time. Any "reopen a saved project and re-render" flow requires a *new* dataset-ref property on the tree; it is not derivable from the current model.

2. **The dispatch predicate is `length(object@blocks) > 0L || object@type %in% c("summary","crosstab","overall")`** (`render_plan.R:66-68`). Note `"param"` is NOT in this literal gate — a param object reaches `.render_ard` only via a block-kind check inside `.build_ard`, so a param object with no block and `type != summary/crosstab/overall` misroutes to the raw SQL leg and errors. Any new object type must be added to *both* the type gate and the block path.

3. **`.object_blocks` legacy role→block synthesizer is live in the compile path** (`render_plan.R:1206-1234`), with its own comment: *"the single transitional concession to pre-block objects ... Remove once every producer emits blocks."* Confirmed verbatim. arframe must emit blocks directly so this is **deleted, not migrated** — otherwise the dual-rail Properties panel has two representations to reconcile and round-trip fidelity depends on which path an object took.

4. **Serialization is a hand-written per-class walker at `schema_version = 4L`** (`aaa_class.R:248`, `serialize.R:9`), min-readable `3L` (`serialize.R:12`). Two hand-synced constants with no compile-time link. `report_from_json` rejects versions outside `[3,4]` with class `explorerengine_error_serialize` (`serialize.R:106-116`). Adding a property = editing encoder + decoder + (maybe) migrator; there is **no round-trip completeness test** tying properties to their codec.

5. **`.migrate_doc` is a no-op version stamp** (`serialize.R:126-129`): `doc$schema_version <- .SCHEMA_VERSION; doc`. Verified. It relies on every change being additive (absent option → NULL → renderer default). The redesign's dataset binding + page level are **non-additive** — this migrator cannot express them as written.

6. **`data_item@format` is serialized RAW** (`serialize.R:227` encode, `:327` decode `format = x$format`), bypassing the `.encode_option_value/.restore_options` named-vector protection that `object`/`block` options get. A named-vector format spec loses its names on round-trip (`auto_unbox=TRUE` drops names) — the exact bug the option layer exists to prevent. Verified.

7. **DuckDB catalog connection lifecycle is app-owned, not model-owned** (`shell.R:66-69`): `ws$catalog <- shiny::reactiveVal(explorerengine::engine_open())`, closed via `shiny::onStop(engine_close(...))`. Deliberately NOT created in `new_workspace()` (`workspace_state.R:70-72`) so unit tests never leak a connection. arpillar must keep `engine_open/engine_close` as the connection boundary and arframe owns the reactiveVal + onStop teardown.

8. **cards/tabular idioms already wired (pinned by probe, not contract):** the % base is POPULATION N via `ard_tabulate(denominator=df)`; the Total column is cards `.overall=TRUE` RELABELED by `pivot_across(overall="Total")` (never synthesized) for summary/param/overall, but **crosstab must synthesize Total** (`.synth_total`) because hierarchical overall fills only the SOC level; `pivot_across` emits NO spanners (headers built by hand off `<var>..<arm>` col names). arpillar inherits a tight coupling to a GitHub-only `Remotes` dep (`vthanik/tabular`) whose behavior is pinned by probe — **a tabular version bump can silently break byte-goldens**.

9. **Figures render on-screen as a STATIC `shiny::renderPlot` bound to `plotOutput`** (`task_canvas.R:285, 626`), NOT ggiraph. Confirmed: zero `ggiraph/girafe` in engine R/ or DESCRIPTION. The CLAUDE.md/CLAUDE.local.md "interactive on screen via ggiraph" story **has no code basis** — the word "interactive" in `export.R:3` is stale. If arframe commits to interactive figures, ggiraph + a girafe path + interactive aes (`tooltip/data_id`) on every `.gg_*` geom are net-new work.

10. **screen==paper is genuine for tables/listings, weaker for figures.** Tables round-trip the identical `tabular` render (HTML on screen via `export_plan`, RTF on paper). Figures share only the ggplot *object* — screen goes through ggplot2's device, paper through `tabular::figure()`; two rasterizers, no golden pins the paper figure against the screen figure. The figure golden gate is a **non-byte smoke assertion** (`file.size>0` + `^\{\\rtf1`, `test-export.R:594-671`); only the SCREEN ggplot has a vdiffr golden.

11. **`object@options` is an untyped grab-bag** (`aaa_class.R:199` `options = S7::class_list`), shaped entirely by `do_commit`'s inline conventions and read back by ad-hoc seeds. The `options$*` key names are simultaneously the **serialize contract AND the byte-golden contract** — any key rename churns every RTF golden. arpillar must pin these keys while arframe rebuilds the UI that writes them.

12. **`report@parameters/@filters/@theme` and `task@task_filters/@task_control` are serialized dead-weight** — round-tripped but consumed nowhere in the render path (only `object@filters` is read, `render_plan.R:85`). Either wire them or drop them; do not carry persisted-but-unconsumed state forward.

13. **Free-text filter predicates are injected verbatim** (`render_plan.R:786`, `sprintf("(%s)", trimws(expr))`, explicitly commented as a ponytail verbatim injection). Structured predicates ARE injection-safe (`dbQuoteLiteral/Identifier`). Acceptable for local single-user; a **hard trust boundary to document** if arframe ever ingests remote/multi-user filter input.

14. **Print/format/`as.*` S7 methods only dispatch in an INSTALLED package after `.onLoad` runs `methods_register()`** (`explorerengine-package.R:12-14`), never under `devtools::load_all`. Any redesign test harness asserting on printed/formatted model output must run against `R CMD INSTALL`, or it gets false greens.

## 4. Inconsistency & tech-debt ledger

Ranked by severity. Fix stance = **fix-in-place** (must be resolved *by* the migration or the port re-imports the bug) vs **separate cleanup** (carry, schedule later).

### High
- **Two sources of truth for "output is configured/runnable":** `.is_configured` in the canvas (`task_canvas.R:436-473`) vs the `fail_run` gates in `do_commit` (`setup_pane.R:4007-4018`). Verified both encode per-type required roles independently; they can drift (canvas shows ready, commit rejects). **Fix-in-place:** consolidate into ONE engine-side predicate in arpillar (e.g. `output_status(object)` returning Ready/Draft/Needs-data/Broken) — the Outline rail's live status column (spec §6) *requires* this single predicate anyway.
- **No two-stage render cache.** `render_plan()` has no memoization (`render_plan.R:65`); the canvas `active_plan` (`task_canvas.R:249`) AND setup_pane's `wide_cols` (`setup_pane.R:1155`) each call `render_plan`+`render_table` independently — the committed ARD is re-derived ≥2× per interaction across separate reactive graphs. **Fix-in-place:** split into an ARD stage (key: dataset+filters+blocks+group) and a display stage (key: display options) in arpillar *before* the UI rebuild. This is the single biggest performance win and a prerequisite for the direct-manipulation UX.
- **`do_commit` is a 942-line monolith with no options schema.** **Fix-in-place** (it *is* the migration crux — see §5).

### Medium
- **Dual `@roles`/`@blocks` representation + `.object_blocks` synth shim** (`aaa_class.R:193-206`, `render_plan.R:1206-1234`). **Fix-in-place:** commit to blocks-only, delete the synth, migrate all producers to emit blocks.
- **No dataset binding on the model.** **Fix-in-place** (schema change, forces the migrator — see §3.1, §5).
- **Spec's `report→page→output` topology does not exist** (`aaa_class.R` is `report→task→object`). **Fix-in-place** (non-additive schema change — see §6).
- **Decimals normalization smeared across ~9 helpers** with shape-sniffing to survive JSON round-trips (`.default`-key discriminator at `render_table.R:157-179`; `.seed_schema_decimals/.materialize_decimals/.is_decimals_spec` in render_plan; `.decimals_named/.expand_scalar_dec/.ensure_pct_dec` in render_table). **Fix-in-place:** one canonical decimals model normalized at the serialize boundary collapses most of it — do it while building the typed options codec, not after.
- **Total-column produced two ways** (pivot `overall=` for summary/param/overall vs `.synth_total` for crosstab), guarded by duplicated `isTRUE(options$total)` checks. **Separate cleanup** — correct today, pinned by goldens; unify opportunistically.
- **`.encode_option_value/.restore_options` bespoke codec, already inconsistently applied** (`data_item@format` skips it, §3.6). **Fix-in-place:** replace with one typed option codec shared by every property — as options grow richer in the Options rail (recodes, per-stat decimals, palettes) this ad-hoc layer is a recurring round-trip-bug source.
- **`do_commit` silently skips malformed rows with `next`** (`setup_pane.R:4293/4611/4681`): recode row with absent column, stack entry with no vars, legend row with absent column — no user feedback. **Fix-in-place** when rebuilding commit: surface dropped rows.
- **`catalog_grid` rebuilds the whole catalog data.frame on every invalidation** (`engine_duckdb.R:190-228`), called raw from 6 sites (`manage_catalog.R:43` reactive, but `manage_sources.R:35`, `manage_viewer.R:29`, `shell.R:355`, `setup_pane.R:460`, `task_canvas.R:547` bypass it). **Fix-in-place:** one shared cached grid reactive keyed on catalog identity.
- **Blocking synchronous import** (`register_dataset` → `artoo::read_* + write_parquet` on the main thread, `engine_duckdb.R:158-179`); a large xpt folder freezes the UI, no progress. **Fix-in-place:** adopt promises/mirai up front — it changes the notification/progress model, so retrofitting later is costly.
- **`do_import` has no error handling around single-file import** (`register_path` only tryCatches the directory loop, `register_path.R:138`); a corrupt single file propagates an uncaught error instead of the honest skipped/duplicates notification. **Fix-in-place.**
- **Error-class granularity is flat** — nearly every compile abort is `explorerengine_error_input` regardless of cause; app aborts are `explorer_error_input`. Section A UI catches neither, so a failed engine import surfaces uncaught. **Separate cleanup** for granularity; **fix-in-place** the missing catch in `do_import`.

### Low
- **Declared keybindings never reach the browser** (`keybindings()` at `commands.R:83-90` is unit-tested but has no runtime caller; `explorer.js:9-18` only binds ⌘K + Escape; `mod+z`/`mod+shift+z` at `shell.R:218,228` are dead declarations). **Fix-in-place:** either emit the keymap to JS or delete the abstraction.
- **Unbounded undo/redo stacks** — each undoable `dispatch` deep-copies the full report onto `ws$undo` (`commands.R:74`), no cap, one snapshot per Run. O(report) memory per action; a leak over a long session. **Fix-in-place:** bounded stack + consider diff-based/coalesced snapshots (spec §9 autosave makes this worse).
- **`column_precision` pulls the ENTIRE column into R** (`dataitems.R:472-492`, `SELECT <col> FROM <view>` no LIMIT/aggregation) — the one dataitems helper that does NOT push down; a per-interaction bottleneck on wide/tall data. **Separate cleanup:** push a `MAX(length(...))`-style precision probe into DuckDB, or carry a stored ADaM precision attribute.
- **`n_distinct` declared but populated nowhere** (grep: only serialize/emit reference it) — dead model + JSON state. **Fix-in-place:** wire the lazy cardinality fill or drop the property.
- **Two owners for "clear selection when dataset disappears"** (`manage_catalog.R:63-68` observer vs `shell.R:337-360` delete command). **Separate cleanup.**
- **Inline-JS `Shiny.setInputValue` selection plumbing** with dataset id interpolated into JS strings (`manage_catalog.R:346-355`, `manage_sources.R:101-105`) — currently safe via `.unique_id` scrub, un-idiomatic. **Fix-in-place** when rebuilding the grid: proper input bindings.

## 5. Migration risk hotspots

The three hardest areas to move to the dual-rail model, in descending order.

**A. `setup_pane.R` `do_commit` + draft sprawl (the crux).** 7085 lines, one module, mixing UI construction, ~30 `draft_*` reactiveVals, ~60 observers, and a 942-line commit that reads every input and hand-shapes one flat options list. The dual rail *splits* Roles/Options/Filters/Ranks into separate panes, but all four currently commit **atomically through one function**. You cannot split the commit per-rail until a declarative options schema + per-rail serializers exist — otherwise the four panes fight over one shared options list. Compounding this:
- The **`suspendWhenHidden=FALSE` "folded panels still commit" contract** (`setup_pane.R:4855-4857`) is load-bearing and browser-only (already in project memory: setup-suspend-contract). arframe rails that lazy-mount or unmount hidden panes will **silently drop config** unless draft state is lifted OUT of the DOM into a structured store first. This is the single most likely source of data-loss regressions.
- **`fresh_after_seed`/`seed_gen`** task-isolation via a manual per-key generation counter (`setup_pane.R:475-495`) is fragile; the clean fix (module-per-object instancing) changes how every draft is seeded and how commit fires.
- **`.seed_roles/.blocks_from_order` round-trip only the one-block-per-kind shape** (`setup_pane.R:5347-5361, 5414-5436`) — a JSON project carrying interleaved same-kind blocks would silently flatten on open+re-commit. The migrator path must be validated against the slot model.

**B. The engine render seam (must be re-shaped before the UI, not after).** The seam is total and correct but monolithic. The dual-rail Options/Ranks panes fire *frequent render-side-only edits* (decimals, cutoff, rank, level hide/relabel, group order, zero) — each currently triggers a full DuckDB collect + cards ARD rebuild because they are all applied downstream of the same `active_plan` reactive. arframe **must land the two-stage cached seam** (ARD stage / display stage) or "screen==paper" direct manipulation feels slow on real clinical datasets. Secondary: the wide display frame is shape-specific (summary/crosstab/overall/param each pivot differently, Total two ways) and downstream shapers key off literal column names (`variable/stat_label/soc/l2../label/row_type/<arms>/Total`) — migrating to a stable arpillar display-frame contract means pinning those names, brittle if roles get renamed. And `classify_cols()` stub detection is a hard-coded name allowlist (`export.R:698-699`) that silently misclassifies any stub column with an unexpected name as an arm — a `render_table` rename in arpillar breaks it with no error, only a golden diff catches it.

**C. The state store + document topology.** The flat `reactiveValues` store (`workspace_state.R:31-69`) with section-scoped pane slots (`pane_left_manage` etc.) is built around the manage/report split and **one-object-per-task**. `active_object(ws)` returns the *first object on the active task* (`workspace_state.R:94-100`) — baked into `.filter_setup_panes`, the statusbar, and setup_pane; `commit_object` appends unconditionally so a second commit creates a hidden object the accessor never surfaces. arframe's report→page→**multiple objects per page** with an Outline rail breaks this: it needs a **selected-object pointer** (`selected_object` field exists but `active_object` ignores it in favor of first-of-task). Because the undo stack stores full report snapshots keyed to the `report→task→object` shape, any document-model change (page level, multi-object pages) ripples through `commit_object/active_object/undo/ensure_active_task` together — they must move as one unit.

## 6. Hardening deltas to the redesign spec

Specific changes the audit forces on `docs/superpowers/specs/2026-07-01-arframe-ui-redesign-design.md`. The spec is strong on IA and paradigm; these are the places where it assumes something the code contradicts or under-specifies.

1. **§9 topology is wrong — the model is `report→task→object`, not `report→page→output`.** The spec asserts `report → page → output → role → dataitem` (lines 158, 43). The actual S7 tree is `report→task→object→role/block→data_item` with **no page level** (`aaa_class.R:243-255,217,193`). **Delta:** the spec must call out that inserting a page level (or renaming task→page + allowing multiple objects per page) is a **non-additive `schema_version 4→5` migration** requiring a real `.migrate_doc` transform chain (the current one is a no-op stamp, `serialize.R:126-129`). This is not a rename; it is the largest single model change and it gates persistence.

2. **§9 must add a dataset-binding property — it does not exist today.** The spec says the JSON model is the "reopenable build" and "the whole set of outputs persists as one project" (lines 157-161), but **no model class carries a dataset id/ref**; `render_plan` takes it as a call argument (`render_plan.R:65`). **Delta:** add a `dataset` (id/ref) property on the object (or page) as an explicit, called-out schema change. Without it, a saved project literally cannot re-render — the spec's core persistence promise is unbacked by the model. This forces the migrator, and touches encode/decode/emit.

## 7. Recommended staged build order

Grounded in the reuse map: migrate the load-bearing spine first (it is `keep`/`adapt` and everything else stands on it), rebuild the app frame last (it is `rebuild` and depends on the hardened engine). Each stage has a check that can actually fail.

**Stage 0 — Package rename + install harness.** `explorerengine→arpillar`, `explorer→arframe`, strip `ex_`. Add `Remotes:` for artoo/datasetviewer/tabular. **Check:** both packages `R CMD INSTALL` clean; existing tests green under installed (not `load_all`) so S7 print/format dispatch is real (§3.14).

**Stage 1 — Engine model hardening (the spine, done before any UI).** In order:
1. **Dataset binding** — add the dataset-ref property to object (or page); thread it so `render_plan` can read it from the model, keeping the call-arg path as a fallback during migration. **Check:** a serialized report re-renders with no external id passed.
2. **Page level** — insert page between report and task-as-object (or rename task→page + allow multiple objects). **Check:** round-trip a two-page, multi-object report through JSON.
3. **Real migrator** — build `.migrate_doc` v4→v5 transform chain for (1)+(2); link `.SCHEMA_VERSION` to the class default (§3.4). **Check:** a v4 fixture opens, migrates, and renders identically; a round-trip completeness test asserts every property survives encode→decode.
4. **Blocks-only** — delete `.object_blocks` synth (`render_plan.R:1206-1234`); make blocks the sole render input. **Check:** all render tests pass with the synth removed (proves no producer relies on it — this is the gate on whether Stage 5 can emit blocks directly).
5. **Typed option codec** — one shared codec for options AND `data_item@format` (kills the §3.6 bypass); one canonical decimals model at the serialize boundary (collapses the §4 decimals sprawl). **Check:** named-vector format + mixed-length options round-trip losslessly.

**Stage 2 — Two-stage cached render seam.** Split `render_plan` into an ARD stage (key: dataset+filters+blocks+group) and a display stage (key: display options); the `render_table.R` shapers become the cached display stage. Add `output_status(object)` as the single configured predicate. **Check:** editing a display-only option (decimals/cutoff/rank) does NOT re-run the DuckDB collect or cards ARD (assert via a call counter); **byte-golden RTF + vdiffr fully re-verified** — this is the highest golden-regression-risk stage.

**Stage 3 — Real reproducible-code emitter.** Rebuild `emit_code` to emit a runnable SQL+cards analysis script (not constructor calls) derived from the same seam; include `object@actions` (fix the §3 round-trip gap). **Check:** the emitted script, run standalone, regenerates the ARD/display frame byte-identical to `render_plan`.

**Stage 4 — Data-mode engine readers + async import.** New column-metadata/value-summary reader for the variable profile; push `column_precision` down; make import non-blocking (promises/mirai). **Check:** import a large xpt folder without freezing (progress observed); profile reader returns type/label/distinct/missingness on a lazy view without materializing the column.

**Stage 5 — arframe shell + dual-rail frame.** Now the engine is stable and blocks-only. Build: the dual-rail frame (rebuild `shell.R/rail.R/appbar`), the five left activities, the right properties rail as **thin per-rail serializers over the Stage-1 typed schema** (this is where the 942-line `do_commit` dies), the canvas shell-preview with ghost slots + direct manipulation, Data mode, JSON persistence UI. Port the command bus (`commands.R`, keep) with a bounded undo stack, and re-encode the keep-mounted/isolate lessons in the new right rail. **Check:** the setup-suspend contract regression test — commit reads a config from a pane the user never opened, with draft state held OUT of the DOM (this is the data-loss gate); a full report saves, closes, reopens, and re-renders every output identically.

**Stage 6 — Interactive figures (if scoped to v1).** Add ggiraph + `render_girafe()` + interactive aes; verify `girafe(ggobj=patchwork)`; add the figure-through-`tabular::figure()` golden (§6 delta 4). **Check:** on-screen figure is interactive AND a golden pins paper vs screen. Defer if v1 ships static figures.

Rationale for order: Stages 1-2 are the `keep`/`adapt` spine and are non-negotiable prerequisites — the UI (Stage 5, all `rebuild`) is unbuildable without the typed options schema (splits the four rails) and the two-stage cache (makes direct manipulation feel instant). Stage 4 unblocks Data mode independently. Stage 6 is the only genuinely net-new subsystem and is safely last (and optional for v1).
